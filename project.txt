================================================================================
PROJECT BLUEPRINT: ZERO-CONFIG LOCAL DNS & SSL ROUTER
================================================================================

1. WHY WE ARE BUILDING THIS
--------------------------------------------------------------------------------
The fundamental problem in local web development is the friction involved in 
routing and securing local environments. Developers frequently need to test 
third-party services (e.g., payment gateway sandboxes, webhooks, OAuth 
callbacks) that strictly require secure HTTPS connections and standard domains 
rather than raw local IPs or localhost ports.

Currently, achieving this involves:
- Manually generating SSL certificates.
- Configuring complex reverse proxies (like NGINX).
- Manually editing system-level files like `/etc/hosts`.

This manual process is tedious, error-prone, and heavily dependent on the host 
operating system. The goal of this project is to eliminate this friction entirely. 
We are building an open-source, cross-platform utility that handles DNS mapping, 
certificate generation, OS trust injection, and traffic proxying with a single 
click, drastically improving the Developer Experience (DX).

2. WHAT WE ARE GOING TO BUILD
--------------------------------------------------------------------------------
We are building a lightweight, cross-platform CLI tool paired with a modern, 
glassmorphism web dashboard. It acts as an automated background reverse proxy 
and local DNS manager.

Key Capabilities:
- Instant Local Domains: Maps custom domains (e.g., `app.test`) to local ports 
  (e.g., `3000`).
- Auto-SSL: Dynamically generates and assigns trusted SSL certificates to those 
  local domains, ensuring a "Green Padlock" in the browser.
- One-Click CORS Bypass: Injects headers to prevent cross-origin resource errors 
  during local API testing.
- Platform Agnostic: Operates seamlessly across Linux, macOS, and Windows.
- Zero Configuration: Driven entirely by a minimalist web UI; no text files or 
  CLI arguments required for daily use.

3. ARCHITECTURAL MODULES & DEVELOPMENT PLAN
--------------------------------------------------------------------------------
The system is divided into six distinct phases. The AI agent should build 
these sequentially, ensuring stability before moving to the next.

PHASE 1: Core Architecture & State Management
Goal: Establish process separation and data persistence.
- Module 1A: CLI Orchestrator. Build the global command interface (e.g., 
  `proxigo ui`) responsible for starting the background daemon.
- Module 1B: The Daemon. Create a persistent background Node.js server to 
  hold routing state and serve the UI.
- Module 1C: State Engine. Implement JSON-based configuration storage in the 
  user's home directory to persist active routes across reboots.

PHASE 2: The Reverse Proxy Engine
Goal: Intercept and route HTTP traffic.
- Module 2A: Port Binding. Securely bind the Daemon to ports 80 and 443 on 
  `127.0.0.1`.
- Module 2B: Dynamic Routing. Parse incoming `Host` headers and proxy 
  requests to the corresponding mapped local port.
- Module 2C: Middleware Injector. Add support for injecting headers, 
  specifically `Access-Control-Allow-Origin: *` for the CORS bypass feature.

PHASE 3: Cross-Platform OS Operations (DNS)
Goal: Safely modify the hosts file across different environments.
- Module 3A (Linux): Script modifications to `/etc/hosts` using `pkexec` or 
  `sudo` wrappers for privilege escalation.
- Module 3B (macOS): Script modifications to `/etc/hosts` utilizing 
  `osascript` for native password prompts.
- Module 3C (Windows): Script modifications to `C:\Windows\System32\drivers\etc\hosts` 
  using PowerShell and the `Start-Process -Verb RunAs` command.

PHASE 4: SSL Generation & OS Trust Store
Goal: Automate the "Green Padlock" locally.
- Module 4A: CA Generation. Use `node-forge` or a wrapper for `mkcert` to 
  generate a local Certificate Authority upon installation.
- Module 4B: Trust Injection. 
    - Linux: Copy CA to `/usr/local/share/ca-certificates/` and run 
      `update-ca-certificates`.
    - macOS: Use `security add-trusted-cert` for the Keychain.
    - Windows: Use `certutil -addstore -f "ROOT"`.
- Module 4C: SNI Handling. Configure the Daemon's HTTPS server to use Server 
  Name Indication to serve the correct SSL certificate based on the requested domain.

PHASE 5: The Web Interface (UI/UX)
Goal: Build the developer-facing control panel.
- Module 5A: Static UI Export. Build a fully client-side Next.js dashboard 
  using Tailwind CSS (Glassmorphism design) and export as static HTML/JS.
- Module 5B: UI Serving. Configure the Daemon to serve these static files 
  on a dedicated port (e.g., `9999`).
- Module 5C: Real-time Communication. Implement Server-Sent Events (SSE) or 
  WebSockets to sync state instantly between the Daemon and the UI.

PHASE 6: Packaging & Distribution
Goal: Make the tool easily installable.
- Module 6A: NPM Configuration. Set the `bin` directive in `package.json` to 
  link the CLI orchestrator globally.
- Module 6B: Post-Install Setup. Write a script to initialize the configuration 
  directory and verify port availability (80/443) upon NPM installation.

FEATURE SPECIFICATION: DYNAMIC WILDCARD SUBDOMAINS (MULTI-TENANT ROUTING)
================================================================================
Feature ID: VR-001
Target: vroute Desktop (Tauri Backend / Proxy Daemon)

1. THE PROBLEM CONTEXT
--------------------------------------------------------------------------------
When engineering a modular property management system or a multi-category 
merchant portal, the architecture often relies on multi-tenancy. Each client, 
property, or business receives a dedicated subdomain (e.g., `hotel-a.app.test`, 
`hotel-b.app.test`, or `merchant.finde.test`). 

Testing this locally usually requires developers to manually add every single 
tenant subdomain to their system's `hosts` file and reverse proxy configuration. 
This breaks the flow of development, especially when dynamically provisioning 
new test merchants or properties.

2. FEATURE OBJECTIVE
--------------------------------------------------------------------------------
Allow developers to register a single wildcard domain (e.g., `*.app.localhost` 
or a managed `*.app.test`) in the vroute dashboard. The proxy will automatically 
intercept any subdomain, route it to the correct local application port, and 
inject identifying headers so the local backend knows exactly which tenant is 
being requested.

3. ARCHITECTURAL WORKFLOW (FOR AI IMPLEMENTATION)
--------------------------------------------------------------------------------

A. The DNS & OS Layer Workaround
Operating system `hosts` files (Windows, macOS, Linux) do NOT natively support 
wildcard entries (like `*.app.test`). To maintain a "zero-config" experience 
without building a complex Port 53 DNS server, the implementation should utilize 
one of two strategies:
   - Strategy 1 (The `.localhost` Standard): Modern browsers automatically 
     resolve any subdomain ending in `.localhost` (e.g., `client.app.localhost`) 
     to 127.0.0.1. vroute can encourage this TLD for multi-tenant setups, 
     requiring zero `hosts` file modification.
   - Strategy 2 (Dynamic Sync): If the developer insists on `.test`, the vroute 
     UI will have a "Tenant Subdomains" input field under the wildcard route. 
     When the developer types `tenant1, tenant2`, the backend instantly appends 
     `tenant1.app.test` and `tenant2.app.test` to the OS `hosts` file.

B. The Proxy Interception & Header Injection Layer
When an HTTP/HTTPS request reaches the vroute proxy:
   1. The proxy parses the incoming `Host` header.
   2. It checks its internal state against registered wildcard patterns 
      (using Regex or glob matching).
   3. If a match is found (e.g., `Host: branch-dhaka.app.localhost`), it extracts 
      the subdomain (`branch-dhaka`).
   4. The proxy seamlessly injects a new HTTP header into the request before 
      forwarding it to the target port (e.g., `3000`). 
      Example injected header: `X-Vroute-Tenant: branch-dhaka`.
   5. The local Next.js or Node.js backend reads this header and dynamically 
      switches the database schema or fetches the specific merchant's data.

C. The SSL Generation Layer
To maintain the "Green Padlock" experience across all tenants without 
generating hundreds of individual certificates:
   - The CA generator module must generate a Subject Alternative Name (SAN) 
     Wildcard Certificate. 
   - When generating the cert for `app.localhost`, the `node-forge` or `rcgen` 
     module must add `*.app.localhost` to the SAN extension.
   - The HTTPS proxy will then serve this single wildcard certificate for any 
     requested subdomain, ensuring seamless secure connections.

4. USER INTERFACE (DASHBOARD) INTEGRATION
--------------------------------------------------------------------------------
- When clicking "Add Route", the developer can toggle a "Multi-Tenant" switch.
- The URL input changes to accept a wildcard format (e.g., `*.myapp.test`).
- A settings panel allows the developer to customize the injected header key 
  (Default: `X-Vroute-Tenant`, Customizable to: `X-Merchant-ID`, etc.).
- The Active Routes grid displays the wildcard route with a visual indicator 
  that it is intercepting dynamic subdomains.
