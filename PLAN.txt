## Phase 7: Dynamic Wildcard Subdomains (VR-001)

### 7.1 Problem Statement

Multi-tenant apps require dedicated subdomains per client (e.g., `hotel-a.app.test`, `merchant-b.app.test`). Currently, developers must manually add each subdomain to `/etc/hosts` and proxy config. This breaks development flow when dynamically provisioning new tenants.

### 7.2 Feature Objective

Allow developers to register a single wildcard domain (e.g., `*.app.localhost`) that:
1. Intercepts any subdomain automatically
2. Routes to the correct local port
3. Injects tenant identity via HTTP headers
4. Serves a wildcard SSL certificate (green padlock)

### 7.3 Design Decisions

#### DNS Strategy (Critical)

**Problem:** `/etc/hosts` does not support wildcards.

**Decision:** Two-tier approach:

| Strategy | TLD | DNS Method | Hosts File | Best For |
|----------|-----|------------|------------|----------|
| **Primary** | `.localhost` | Browser auto-resolves to 127.0.0.1 (RFC 6761) | Not needed | Most use cases |
| **Fallback** | `.test` | Local DNS server on port 53 | Not needed | When `.localhost` unavailable |

**Why NOT dynamic hosts sync:**
- Requires sudo per tenant (bad UX)
- No true wildcard support
- Race conditions with concurrent adds

**Why a local DNS server:**
- True wildcard resolution (`*.app.test → 127.0.0.1`)
- No hosts file modifications needed
- Single process, clean shutdown
- Works on all platforms

#### Route Storage Format

```json
{
  "routes": {
    "app.localhost": {
      "port": 3000,
      "ssl": true,
      "cors": true,
      "wildcard": true,
      "tenantHeader": "X-Vroute-Tenant"
    },
    "api.localhost": {
      "port": 3001,
      "ssl": true,
      "cors": false
    }
  }
}
```

**Key additions to `RouteConfig`:**
- `wildcard: boolean` — enables subdomain matching
- `tenantHeader: string` — customizable header key (default: `X-Vroute-Tenant`)

#### Proxy Matching Algorithm

```
resolveRoute(host, routes):
  1. Exact match: routes[host] → return if found
  2. Wildcard match: for each route where route.wildcard === true:
     - Extract pattern base: "*.app.localhost" → "app.localhost"
     - Check if host ends with ".{base}" or equals "{base}"
     - If match: return route + extracted subdomain
  3. No match → return undefined
```

**Shared helper function** replaces 3 duplicate lookups in `proxy.ts` (lines 24, 80, 103).

#### SSL Wildcard Certificate

- Generate cert with SAN: `*.app.localhost` (wildcard)
- Single cert covers all subdomains
- Cache by wildcard pattern, not individual subdomain
- SNI callback resolves wildcard routes before cert lookup

#### Header Injection

When a wildcard route matches:
1. Extract subdomain from host (e.g., `branch-dhaka.app.localhost` → `branch-dhaka`)
2. Inject header: `X-Vroute-Tenant: branch-dhaka`
3. Backend reads header to identify tenant

### 7.4 Implementation Plan

#### Step 1: Config Schema Update

**File:** `packages/vroute-cli/src/state/config.ts`

- Add `wildcard?: boolean` and `tenantHeader?: string` to `RouteConfig`
- No breaking changes — new fields are optional

#### Step 2: Route Resolution Helper

**File:** `packages/vroute-cli/src/daemon/proxy.ts`

Create shared function:
```typescript
interface ResolvedRoute {
  config: RouteConfig;
  subdomain?: string;  // extracted if wildcard match
}

function resolveRoute(host: string, routes: Record<string, RouteConfig>): ResolvedRoute | undefined
```

Replace all 3 exact lookups with this function.

#### Step 3: Local DNS Server

**New file:** `packages/vroute-cli/src/dns/server.ts`

- Use `dns2` package or raw `dgram` socket
- Listen on port 53 (requires sudo)
- Resolve `*.configured-wildcard.com` → 127.0.0.1
- Start/stop with daemon lifecycle
- Fallback: if port 53 unavailable, log warning and skip

**Dependencies:** Add `dns2` to package.json

#### Step 4: SSL Wildcard Certs

**File:** `packages/vroute-cli/src/ssl/certs.ts`

- Modify `generateDomainCert()` to detect `*.` prefix
- Add SAN: `{ type: 2, value: '*.app.localhost' }`
- Cache certs by wildcard pattern

**File:** `packages/vroute-cli/src/daemon/https.ts`

- Update `SNICallback` to use `resolveRoute()` helper
- Generate wildcard cert on first match, cache it

#### Step 5: Header Injection

**File:** `packages/vroute-cli/src/daemon/proxy.ts`

In `proxyRes` handler:
- If resolved route has `subdomain` (wildcard match)
- Set header: `res.setHeader(route.tenantHeader, subdomain)`

#### Step 6: API Updates

**File:** `packages/vroute-cli/src/daemon/server.ts`

- `POST /api/routes`: Accept `wildcard` and `tenantHeader` fields
- `DELETE /api/routes/:domain`: Handle wildcard cleanup
- Skip hosts file worker for wildcard routes

#### Step 7: CLI Updates

**File:** `packages/vroute-cli/src/cli/index.ts`

- `add` command: detect `*.` prefix → set `wildcard: true`
- `list` command: show wildcard indicator
- `remove` command: handle wildcard cleanup

#### Step 8: Dashboard Updates

**File:** `packages/vroute-cli/src/dashboard/src/App.tsx`

- Add "Multi-Tenant" toggle to add route form
- Show wildcard icon on wildcard routes
- Customize header key input (visible when multi-tenant enabled)
- Skip external link for wildcard routes (no single URL)

### 7.5 Security Considerations

| Risk | Mitigation |
|------|------------|
| Wildcard routes exposed to network | Always bind proxy to 127.0.0.1 only |
| DNS server on port 53 intercepts all queries | Only respond for configured wildcards, forward rest |
| Header injection spoofing | Headers are injected by proxy only, not from client |
| Wildcard cert misuse | Certs are local-only, CA is self-signed |

## Best Practices

### Code
- Extract shared helpers (like `resolveRoute()`) instead of duplicating logic
- Optional fields in config (`wildcard?: boolean`) for backward compatibility
- Graceful degradation (if DNS server can't bind port 53, skip wildcard DNS)

### Security
- Never expose proxy to 0.0.0.0 — always 127.0.0.1
- Local-only SSL certificates (not for production)
- Sanitize domain inputs (no shell injection in sudo calls)

### UX
- `.localhost` as default TLD (zero config)
- Clear visual indicators for wildcard routes
- Meaningful error messages when ports are unavailable

### Distribution
- Keep npm package small (exclude dev files)
- Version bumps: patch for fixes, minor for features, major for breaking changes
- Test locally before publish (`sudo npm install -g .`)
